# Spec S1.1 Explanation: Project Structure + Dependency Declaration

## Why This Spec Exists

Every Python project needs a properly configured foundation before any code is written. S1.1 establishes the project skeleton so that all subsequent specs (S1.2–S9.6) have a consistent, tooled environment. Without this, developers would encounter import errors, inconsistent formatting, missing dependencies, and configuration drift.

## What It Does

1. **`pyproject.toml`** — Single source of truth for project metadata, all 30+ dependencies (pinned with minimum versions), and tool configurations (ruff, pytest, coverage). Uses PEP 621 standard with hatchling build backend.

2. **`.gitignore`** — Prevents secrets (`.env`), large data files (`data/`, ~17GB), build artifacts, IDE files, Docker volumes, and Node modules from being committed.

3. **`.env.example`** — Documents all 17 environment variables the system needs: 5 LLM API keys (DeepSeek, Qwen, Kimi, Groq, Google), 3 Indian govt API keys (Bhuvan, FIRMS, data.gov.in), 3 Bhashini translation keys, Ollama host, database passwords, and Langfuse secrets. All values are empty — real keys go in `.env` (gitignored).

4. **Python package structure** — 19 `__init__.py` files across `src/` and `tests/` making all directories importable Python packages. Matches the directory layout specified in `design.md`.

5. **Test infrastructure** — `tests/conftest.py` with shared fixtures, `pytest-asyncio` configured for auto mode, coverage targeting `src/`.

## How It Works

- `uv sync --extra dev` reads `pyproject.toml` and installs everything in a `.venv`
- `uv run pytest` discovers tests in `tests/` (configured in `[tool.pytest.ini_options]`)
- `uv run ruff check src/ tests/` lints with line-length=100, Python 3.11 target
- All 19 `__init__.py` files are empty — they just mark directories as Python packages

## Interview Talking Points

- **Why `pyproject.toml` over `setup.py` + `requirements.txt`?** — PEP 621 standard, single file for metadata + deps + tool config, no arbitrary code execution during install
- **Why `uv` over `pip` + `venv`?** — 10-100x faster dependency resolution (Rust-based), deterministic lockfile, replaces pip + pip-tools + virtualenv in one tool
- **Why pin with `>=` instead of `==`?** — Allows compatible updates while setting a minimum quality floor. `uv.lock` (generated by `uv sync`) provides exact reproducibility
- **Why `hatchling` build backend?** — Lighter than setuptools, native PEP 621 support, no setup.py needed
